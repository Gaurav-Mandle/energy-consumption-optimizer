Getting Started

Introduction

What is this project about?

A smart-city project needs to forecast how much electricity a neighborhood will use, hour by hour. Knowing this in advance lets the power grid supply exactly what is needed, avoiding waste and preventing shortages.

We built a Machine Learning model that predicts electricity load using simple, everyday signals — the time of day, the temperature, the humidity, and whether it is a weekend.

Why This Matters

The Problem

Real-world energy data is messy — and hard for machine learning models to use directly:

  • Missing Sensor Readings: Sensors occasionally fail, leaving gaps in temperature and humidity data.
  • Duplicate Records: The same reading sometimes gets logged twice, distorting overall patterns.
  • Mismatched Scales: Hour ranges 0-23, temperature 5-45, humidity 40-90. Models get confused by the size difference.
  • No Live Prediction System: There was no way for the city to get an instant load forecast on demand.

Before We Dive In

Key Concepts

Understanding these foundational machine learning terms explains how the model learns from our smart city dataset.

Regression
Predicting a continuous numerical value (like kWh electricity load) rather than classifying data into labels (like Yes/No).
Linear Regression
Finds the best straight-line formula mapping inputs to predictions:
load = w1*(hour) + w2*(temp) + ... + bias.
Train / Test Split
Dividing data into 80% for training (model learns from it) and 20% for testing (model is evaluated on unseen data to test real performance).
Feature Scaling
Brings all features (hour, temp, humidity) to the same range so that the mathematical equations treat each variable equally.

Evaluation Metrics

How We Judge the Model

After training, we use standard regression metrics to tell us exactly how close the predictions are to the actual grid values.

MSE (Mean Squared Error)

Average of all (actual - predicted)² errors. It square-punishes large errors heavily, serving as a clear yardstick for predicting energy spikes. Lower is better.

RMSE (Root Mean Squared Error)

Square root of MSE. This brings the error metric back to the original unit (kWh), making it easy to interpret (e.g. "predictions deviate by ±X kWh on average").

R² (Coefficient of Determination)

Explains the percentage of variations in the electricity load captured by the model. 100% (1.0) is a perfect fit, while 0% means the model is as bad as guessing the average.

Core Focus

Feature Scaling Deep Dive

Our features operate on wildly different scales: Hour (0-23), Temp (5-45°C), Humidity (40-90%). Without scaling, the model bias-weights columns with larger numerical ranges.

StandardScaler

Centers data around 0 with standard deviation of 1. Keeps outliers balanced.

MinMaxScaler

Squeezes all values strictly between 0 and 1. Preserves bounding limits.

💡 Scaling Rule
Fit your scaler only on the training set (80%) to calculate mean/scale, then transform both train and test sets to prevent data leakage.

Why is the R² 1.92%? Since electricity load generation depends on non-linear factors (like sinusoidal time waves and V-shaped temperature deviations), a simple straight-line Linear Regression model yields a low R² score. This highlights the need for non-linear models or non-linear feature engineering in a real grid.

Summary

Conclusion & Takeaways

We successfully constructed an end-to-end regression pipeline mapping smart city electricity usage features to demand load.

  • Data Quality First: Cleaned 140 missing sensor entries and dropped 30 duplicate logs before training.
  • Scaling Necessity: Feature scaling is crucial. Without standardizing values, the model gets biased towards features with larger scales.
  • Model Limitations: The low R² score of 1.92% demonstrates that simple Linear Regression cannot represent complex sinusoidal daily variations and temperature fluctuations.
  • Serialized Delivery: Saved our finalized linear model and StandardScaler coefficients to disk using joblib.
  • Cloud Path: Provided a complete architecture outlining Flask wrapper microservices for AWS SageMaker or EC2 grid integration.
⚡ Final Verdict
This project successfully establishes the data cleaning, standardization, evaluation, and serialization steps required to deliver an operational machine learning service.

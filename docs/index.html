Project Overview

In modern smart cities, balancing electricity supply and demand is critical. Producing too much power results in waste; producing too little causes blackouts.

This project builds an end-to-end Machine Learning pipeline that forecasts energy demand (electricity load) on an hourly basis. By predicting load ahead of time, city operators can adjust grid generation levels dynamically.

Why This Matters

Real-world Data Challenges

Raw environmental and load data logged by smart sensors is rarely clean or ready for modeling.

  • Sensor Outages: Network drops cause missing values in temperature and humidity readings.
  • Logging Glitches: Sensor duplicates record identical timestamps multiple times.
  • Scale Imbalances: Feature magnitudes differ wildly (e.g. Hour: 0–23, Temp: 5–45, Humidity: 40–90). Larger values dominate calculations incorrectly.
  • API Gap: The grid lacked a real-time prediction interface.

Exploring the Data

Exploratory Data Analysis (EDA)

Analyzing electricity consumption cycles and patterns prior to training is vital.

  • Diurnal Rhythm: Electricity consumption follows a clear daily pattern, peaking during daytime and early evening hours, and dropping significantly at night (lowest around 3 AM - 5 AM).
  • Weekly Patterns: Weekends feature a noticeable drop in overall electricity load (approx 30 kWh lower) due to offices and industrial sectors being closed.
  • Weather Impact: High temperatures correspond to load surges, reflecting cooling demands.

Core Focus

Feature Scaling Deep Dive

Features have very different numerical ranges: hour (0–23), temperature (5–45), and humidity (40–90). Without scaling, algorithms like Linear Regression get biased towards larger magnitude features.

StandardScaler

Transforms features to have a mean of 0 and a standard deviation of 1. Ideal for algorithms assuming normal distribution.

MinMaxScaler

Compresses features to fit strictly between 0 and 1. Preserves original bounds and zero values.

💡 Lab Note: Scaling Rule
Never fit your scaler on test data! Always call fit_transform() on the training set, and only transform() on the test set to avoid data leakage.

Model Training

Linear Regression vs. Random Forest

Actual vs. Predicted Loads & Performance Chart

We trained both models on the scaled features. The evaluation reveals an important ML concept:

Model MSE R² Score Performance Explanation
Linear Regression 5,424.45 1.92% Weak fit. Straight line cannot model cyclical sine waves.
Random Forest 491.14 91.12% Excellent fit. Capture non-linear daily and temp cycles.

Why the R² gap? The electricity load changes non-linearly over the day (sine wave) and temperature (V-shape absolute demand). Random Forest splits the data into small segments to capture these curves perfectly.

Summary

Conclusion & Takeaways

This smart-city energy regression case study highlights fundamental machine learning principles.

  • Garbage In, Garbage Out: Properly cleaning the data (imputing sensor missing values and deleting duplicate entries) is mandatory before modeling.
  • Linearity Assumptions: Linear Regression failed (R² 1.92%) because electricity usage cycles are cyclical (sine wave) and temperature requirements are quadratic.
  • Tree Ensemble Strengths: Random Forest excelled (R² 91.12%) because decision splits can represent non-linear relationships.
